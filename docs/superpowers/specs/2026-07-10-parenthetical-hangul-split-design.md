# 여러 토큰 괄호 가사의 한글 분할 수정 설계

## 배경

Musixmatch richsync 가사는 괄호 구간을 여러 토큰으로 나눠 전달할 수 있다. 예를 들어
`"(이"`, `"밤을"`, `"새워)"`는 하나의 코러스 구간이지만, 현재
`splitHangulSyllables`는 각 토큰을 독립적으로 검사한다. 그 결과 중간 토큰
`"밤을"`만 `"밤"`, `"을"`로 분할되고 두 번째 글자에는 `isPartOfWord: true`가
설정된다.

렌더러는 `isPartOfWord: true`인 토큰의 괄호 파싱을 건너뛰므로 이 글자에서 괄호
상태가 종료된다. 이후 `"새워)"`는 일반 가사로 처리되어 다음과 같은 잘못된 결과가
생긴다.

- 메인: `너와 나 둘이 이 밤을 새워 을 새워)`
- 코러스: `이 밤`

캐시에는 한글 분할 전 문서가 저장되므로 캐시 데이터 자체는 정상이며, 문제는
`LyricsService`가 렌더링 직전에 한글 분할을 적용하는 과정에서 발생한다.

## 목표

- 여러 공급자 토큰에 걸친 둥근괄호 구간을 한글 글자 단위 분할에서 제외한다.
- 괄호 밖의 한글 단어 분할, 타이밍 분배 및 `isPartOfWord` 동작은 그대로 유지한다.
- 기존 괄호 렌더링과 번역 표시 로직은 변경하지 않는다.
- 실제 문제 입력이 메인 `너와 나 둘이 이 밤을 새워`, 코러스 `이 밤을 새워`로
  렌더링되는 회귀 테스트를 추가한다.

## 비목표

- 괄호 내부 가사를 글자 단위로 다시 분할하지 않는다.
- 렌더러의 괄호 레이아웃 또는 코러스 스타일을 변경하지 않는다.
- 대괄호를 코러스 표기로 새로 해석하지 않는다.
- 공급자 데이터 스키마나 캐시 형식을 변경하지 않는다.

## 검토한 접근법

### 1. 한글 분할 단계에서 괄호 범위를 추적한다 — 채택

각 `SyllableVocal`을 순회하면서 괄호 안/밖 상태를 유지한다. 현재 토큰이 괄호 안에서
시작하거나 괄호 문자를 포함하면 해당 토큰을 분할하지 않는다. 괄호 밖의 토큰만
기존 `splitSyllable` 함수로 보낸다.

이 방식은 괄호 토큰을 분할하지 않도록 한 기존 의도를 여러 토큰 구간까지 확장하며,
렌더러와 타입을 변경하지 않아 영향 범위가 가장 작다.

### 2. 렌더러에서 `isPartOfWord` 토큰의 괄호 상태를 유지한다 — 미채택

이미 괄호 안인 경우에는 `isPartOfWord: true`여도 괄호 파서를 통과시키는 방식이다.
코러스 안에서도 글자 단위 진행 효과를 유지할 수 있지만, continuation 토큰 내부의
괄호를 의도적으로 인라인 유지하는 기존 동작과 결합 규칙이 복잡해진다.

### 3. 음절 타입에 괄호 메타데이터를 추가한다 — 미채택

분할 전에 각 토큰에 코러스 여부를 기록할 수 있으나, 타입·캐시·공급자·렌더러 전반을
변경해야 하므로 이번 버그 범위에 비해 과도하다.

## 상세 설계

`splitVocal`을 상태 없는 `flatMap(splitSyllable)` 호출에서 상태를 가진 순회로
변경한다.

1. 보컬마다 `isInsideParenthetical`을 `false`로 초기화한다.
2. 렌더러와 동일하게 `romanizedText ?? text`를 괄호 검사 문자열로 사용한다.
3. 토큰 처리 전에 괄호 안이었거나 토큰 자체에 여는/닫는 둥근괄호가 있으면 원본
   `Syllable`을 그대로 보존한다.
4. 토큰 문자를 순회하며 괄호 밖에서 `(`를 만나면 괄호 안 상태로, 괄호 안에서
   `)`를 만나면 괄호 밖 상태로 전환한다.
5. 괄호 바깥이며 괄호 문자가 없는 토큰만 기존 `splitSyllable`로 분할한다.
6. 보컬이 끝나면 상태를 폐기하여 다음 보컬 줄로 전파하지 않는다.

상태 전이는 `parseWordLevelParentheticals`와 동일한 ASCII 둥근괄호 불리언 규칙만
사용한다. 괄호 안의 추가 `(`와 괄호 밖의 `)`는 상태를 바꾸지 않는 일반 문자로
취급한다. 닫히지 않은 `(`는 해당 보컬의 나머지 토큰을 괄호 안으로 유지한다. 전각
괄호와 중첩 괄호에 새로운 의미를 추가하지 않으며, 기존 정규화와 렌더러 계약을 그대로
따른다.

## 데이터 흐름

```text
Musixmatch/캐시 원본
  -> normalizeLyrics
  -> splitHangulSyllables
       괄호 밖: 기존 글자 단위 분할
       괄호 안: 원본 토큰 보존
  -> buildSyllableRows
       기존 괄호 상태 머신이 메인/코러스를 분리
```

## 테스트 전략

### 단위 테스트

`tests/lyrics/splitHangulSyllables.test.ts`에 다음을 검증한다.

- `"(이"`, `"밤을"`, `"새워)"`가 모두 원본 그대로 유지된다.
- 같은 보컬에서 괄호 앞뒤의 일반 한글 단어는 계속 글자 단위로 분할된다.
- 닫는 괄호 뒤에서 괄호 상태가 해제되어 이후 일반 한글 단어의 분할이 재개된다.
- 단일 토큰 괄호와 기존 구두점·로마자·짧은 토큰 동작은 기존 테스트로 보존한다.

### 렌더링 회귀 테스트

실제 문제 음절 배열에 `splitHangulSyllables`를 적용한 뒤 `buildSyllableRows`를 호출해
다음을 검증한다.

- 메인 텍스트: `너와 나 둘이 이 밤을 새워`
- 코러스 텍스트: `이 밤을 새워`
- 닫는 괄호가 메인 텍스트에 남지 않는다.

### 전체 검증

- `npm run typecheck`
- `npm run lint`
- `npm run test`
- `npm run build`

## 호환성과 위험

변경은 런타임에서 생성되는 표시용 `SyllableLyrics`에만 적용되며 캐시 형식은 바뀌지
않는다. 주요 위험은 괄호 뒤의 일반 토큰까지 분할이 중단되거나 분할기와 렌더러의 상태
전이가 달라지는 것이다. 동일 보컬에서 닫는 괄호 뒤 일반 한글 단어가 다시 분할되는
테스트와 렌더링 회귀 테스트로 이를 방지한다. 렌더러와 동일한 ASCII 불리언 상태 규칙을
사용하고 렌더러 코드는 수정하지 않으므로, 번역 표시, pseudo-karaoke의 `isPartOfWord`
그룹화, 배경 보컬 렌더링에는 직접적인 변경이 없다.
